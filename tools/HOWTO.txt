UNIVERSAL CONVENTIONS · CONVENÇÕES UNIVERSAIS
================================================================================

[EN]
Every tool here follows the same conventions. Memorise these once and they
apply to all 22 tools:

  --lang {en,pt}     Output language. Default: en. 'pt' is European
                     Portuguese (PT-PT, not BR-PT).
  --json             Machine-readable output (every label, risk, fix is
                     keyed in JSON). Pipes well into jq.
  --version          Print version and exit. Centralised in _lib.TOOLS_VERSION.
  --user-agent STR   Override the outbound User-Agent (networked tools).
                     Default is "<tool>/<version> (+ciberacaro.github.io)".
  -                  Anywhere a single positional argument is expected
                     (URL, domain, hash, JWT, hex string), pass '-' to read
                     it from stdin. Enables piping: `... | tool -`.

  Exit codes (same everywhere):
    0   Success — no issues, nothing notable found
    1   Issues found (missing header, expired cert, alg:none, etc.)
    2   Usage / argument error (bad URL, empty input, invalid format)
    3   Network, DNS, or TLS error

  Python 3.8+ standard library only. No `pip install` required.

[PT]

os 22 tools:

  --lang {en,pt}     Língua do output. Por omissão: en. 'pt' é português
                     europeu (PT-PT, não BR-PT).
  --json             Output legível por máquina (cada etiqueta, risco,
                     correção tem chave em JSON). Combina bem com jq.
  --version          Imprime versão e sai. Centralizado em _lib.TOOLS_VERSION.
  --user-agent STR   Sobrepõe o User-Agent enviado (tools que fazem rede).
                     Por omissão: "<tool>/<version> (+ciberacaro.github.io)".
  -                  Onde se espera um argumento posicional único (URL,
                     domínio, hash, JWT, hex), passar '-' lê de stdin.
                     Permite pipes: `... | tool -`.

  Códigos de saída (iguais em todos):
    0   Sucesso — sem problemas, nada de notável
    1   Problemas encontrados (header em falta, cert expirado, alg:none, etc.)
    2   Erro de utilização (URL inválido, input vazio, formato errado)
    3   Erro de rede, DNS, ou TLS

  Python 3.8+, apenas standard library. Sem `pip install`.

1.1 check_headers.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Audit the security-relevant HTTP response headers of a URL: HSTS, CSP,
    X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-
    Policy, the three cross-origin headers (COOP/COEP/CORP), plus info-
    disclosure headers (Server, X-Powered-By). Each missing or weak header
    is reported with a concrete attack/risk and a one-line fix. Detects
    redirect chains and warns when the headers describe the destination
    rather than the origin you asked about.

[EN] QUICK START
    $ tools/check_headers.py https://example.com

[EN] EXAMPLES
    # Scenario 1 — baseline audit of a target site
    $ tools/check_headers.py https://ciberacaro.github.io
    Expected output (truncated):
        URL: https://ciberacaro.github.io
        Status: 200
        Security headers:
          ✓ Strict-Transport-Security: max-age=31536000; ...
          ✗ Content-Security-Policy: (missing)
          ! X-Frame-Options: SAMEORIGIN
          ...
        Score: 5/9 headers OK
        Issues found (4):
          ✗ Content-Security-Policy missing
            Risk: no defense-in-depth against XSS injection.
            Fix:  add `Content-Security-Policy: default-src 'self'; ...`

    # Scenario 2 — machine-readable for piping
    $ tools/check_headers.py https://example.com --json | jq '.issues[].name'

    # Scenario 3 — PT-PT report ready to paste into a writeup
    $ tools/check_headers.py https://example.com --lang pt --no-color

[EN] FLAGS
    Beyond the universals (--lang, --json, --version, --user-agent):
    --timeout SECONDS   Default 10. Raise for slow CDNs or laggy targets.
    --no-color          Strip ANSI codes. Useful for writeups and pipes.

[EN] EXIT CODES
    0  All checked headers OK
    1  Missing or weak headers, or info-disclosure detected
    2  Bad URL (no scheme, malformed)
    3  Network, DNS, or TLS error

[EN] TIPS
    - Score only counts security headers; info-disclosure (Server,
      X-Powered-By) is reported but doesn't lower the score.
    - On 301/302/303/307/308, the redirect chain is shown and a warning
      reminds you the headers describe the destination, not the origin.
    - Pair with header_diff to track regressions when site config or CDN
      changes silently strip a header.

[PT] PROPÓSITO
    Audita os headers HTTP de segurança de um URL: HSTS, CSP, X-Frame-
    Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy,
    os três headers cross-origin (COOP/COEP/CORP), mais headers de
    information disclosure (Server, X-Powered-By). Cada header em falta
    ou fraco é reportado com o ataque/risco concreto e uma correção em
    uma linha. Deteta cadeias de redirect e avisa quando os headers
    descrevem o destino e não o URL original.

[PT] ARRANQUE RÁPIDO
    $ tools/check_headers.py https://example.com

[PT] EXEMPLOS
    # Cenário 1 — auditoria base a um site alvo
    $ tools/check_headers.py https://ciberacaro.github.io
    Output esperado (truncado):
        URL: https://ciberacaro.github.io
        Estado: 200
        Headers de segurança:
          ✓ Strict-Transport-Security: max-age=31536000; ...
          ✗ Content-Security-Policy: (em falta)
          ! X-Frame-Options: SAMEORIGIN
          ...
        Pontuação: 5/9 headers OK
        Problemas encontrados (4):
          ✗ Content-Security-Policy em falta
            Risco:    sem defesa em profundidade contra injeção XSS.
            Correção: adicionar `Content-Security-Policy: default-src ...`

    # Cenário 2 — output legível por máquina
    $ tools/check_headers.py https://example.com --json | jq '.issues[].name'

    # Cenário 3 — relatório em PT-PT pronto para um writeup
    $ tools/check_headers.py https://example.com --lang pt --no-color

[PT] FLAGS
    Para além das universais (--lang, --json, --version, --user-agent):
    --timeout SEGUNDOS  Por omissão 10. Aumentar para CDNs lentas.
    --no-color          Remove códigos ANSI. Útil para writeups e pipes.

[PT] CÓDIGOS DE SAÍDA
    0  Todos os headers verificados OK
    1  Headers em falta/fracos, ou information disclosure
    2  URL inválido (sem esquema, malformado)
    3  Erro de rede, DNS ou TLS

[PT] DICAS
    - A pontuação só conta headers de segurança; information disclosure
      (Server, X-Powered-By) é reportado mas não baixa a pontuação.
    - Em 301/302/303/307/308, a cadeia de redirects é mostrada e há um
      aviso a lembrar que os headers descrevem o destino, não a origem.
    - Combina com header_diff para apanhar regressões quando alterações
      ao site ou CDN removem silenciosamente um header.

1.2 cookie_check.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Analyze every Set-Cookie response header for security flags: HttpOnly,
    Secure, SameSite, Domain, Path, Expires/Max-Age, Priority, and the
    __Host- / __Secure- prefix rules from RFC 6265bis. Flags session-like
    cookies without HttpOnly (XSS amplifier), cookies without Secure on
    HTTPS (MITM target), missing/weak SameSite (CSRF surface), broad Domain
    scope, and cookies that live longer than 1 year.

[EN] QUICK START
    $ tools/cookie_check.py https://example.com

[EN] EXAMPLES
    # Scenario 1 — audit a login page's cookies
    $ tools/cookie_check.py https://target.example/login
    Expected output (truncated):
        URL: https://target.example/login
        Cookies set: 3
          - PHPSESSID
              HttpOnly: ✗ missing       Secure: ✓        SameSite: Lax
              Domain:   target.example   Path: /          Max-Age: session
          - __Host-csrf
              HttpOnly: ✓               Secure: ✓        SameSite: Strict
              ...
        Issues found (2):
          ✗ PHPSESSID lacks HttpOnly
            Risk: any reflected/stored XSS can read the session token.
            Fix:  add `HttpOnly` attribute on the cookie.

    # Scenario 2 — JSON output for a script
    $ tools/cookie_check.py https://target.example --json | jq '.cookies'

    # Scenario 3 — bilingual report into a writeup
    $ tools/cookie_check.py https://target.example --lang pt

[EN] FLAGS
    Beyond the universals: only --timeout SECONDS (default 10).

[EN] EXIT CODES
    0  All cookies OK or no Set-Cookie headers
    1  Issues found
    2  Bad URL
    3  Network error

[EN] TIPS
    - The __Host- prefix is the strongest cookie marker: requires Secure,
      no Domain attribute, and Path=/. Browsers reject violations.
    - SameSite=None must come with Secure or browsers reject the cookie.
    - For a real login flow, follow redirects manually (curl -L) and run
      the tool on each step — auth cookies are often set after redirects.

[PT] PROPÓSITO
    Analisa cada header Set-Cookie e as suas flags de segurança: HttpOnly,
    Secure, SameSite, Domain, Path, Expires/Max-Age, Priority, e as regras
    de prefixo __Host- / __Secure- do RFC 6265bis. Assinala cookies de
    sessão sem HttpOnly (amplificador de XSS), cookies sem Secure em HTTPS
    (alvo de MITM), SameSite em falta/fraco (superfície de CSRF), escopo
    de Domain demasiado amplo, e cookies que duram mais de 1 ano.

[PT] ARRANQUE RÁPIDO
    $ tools/cookie_check.py https://example.com

[PT] EXEMPLOS
    # Cenário 1 — auditar cookies de uma página de login
    $ tools/cookie_check.py https://target.example/login
    Output esperado (truncado):
        URL: https://target.example/login
        Cookies definidos: 3
          - PHPSESSID
              HttpOnly: ✗ em falta     Secure: ✓        SameSite: Lax
              Domain:   target.example  Path: /          Max-Age: sessão
          - __Host-csrf
              HttpOnly: ✓              Secure: ✓        SameSite: Strict
              ...
        Problemas encontrados (2):
          ✗ PHPSESSID sem HttpOnly
            Risco:    qualquer XSS refletido/persistido lê o token de sessão.
            Correção: adicionar o atributo `HttpOnly` no cookie.

    # Cenário 2 — output JSON para um script
    $ tools/cookie_check.py https://target.example --json | jq '.cookies'

    # Cenário 3 — relatório bilingue para um writeup
    $ tools/cookie_check.py https://target.example --lang pt

[PT] FLAGS
    Para além das universais: apenas --timeout SEGUNDOS (por omissão 10).

[PT] CÓDIGOS DE SAÍDA
    0  Todos os cookies OK ou sem Set-Cookie
    1  Problemas encontrados
    2  URL inválido
    3  Erro de rede

[PT] DICAS
    - O prefixo __Host- é o marcador mais forte: requer Secure, sem
      atributo Domain, e Path=/. Browsers rejeitam violações.
    - SameSite=None tem de vir com Secure ou os browsers rejeitam.
    - Para um fluxo de login real, segue redirects manualmente (curl -L)
      e corre o tool em cada passo — cookies de auth são definidos a
      seguir a redirects.

1.3 cors_check.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Probe a URL with several Origin headers and inspect the
    Access-Control-Allow-* response headers. Sends both GET and OPTIONS
    (preflight) requests with arbitrary attacker origins, the magic 'null'
    origin, prefix/suffix tricks (`target.example.attacker.com`,
    `attacker.com.target.example`), and a legitimate origin. Flags
    reflected origins, wildcard ACAO with credentials, accepted 'null'
    origin, and naive prefix/suffix matching bugs.

[EN] QUICK START
    $ tools/cors_check.py https://api.target.example/users/me

[EN] EXAMPLES
    # Scenario 1 — test if an API endpoint reflects arbitrary origins
    $ tools/cors_check.py https://api.target.example/users/me
    Expected output (truncated):
        Probes:
          Origin: https://attacker.example
            GET     → ACAO: https://attacker.example  ACAC: true   [✗ REFLECTED]
            OPTIONS → ACAO: https://attacker.example  ACAM: GET,POST,DELETE
          Origin: null
            GET     → ACAO: null   ACAC: true                       [✗ NULL OK]
          ...
        Issues found (2):
          ✗ Reflected arbitrary Origin with credentials
            Risk: any site can read this endpoint's response with the user's
                  cookies. CSRF + data theft.
            Fix:  validate Origin against an allowlist server-side.

    # Scenario 2 — JSON for a recon report
    $ tools/cors_check.py https://api.example.com/me --json

    # Scenario 3 — PT-PT for a writeup
    $ tools/cors_check.py https://api.example.com/me --lang pt

[EN] FLAGS
    Beyond the universals: only --timeout SECONDS (default 10).

[EN] EXIT CODES
    0  No risky CORS behaviour detected
    1  Reflected origins, wildcard+creds, null accepted, etc.
    2  Bad URL
    3  Network error

[EN] TIPS
    - Run against authenticated endpoints (after login) — CORS misconfigs
      are most damaging when the endpoint requires credentials.
    - Wildcard ACAO ('*') without credentials is often intentional and
      harmless (public APIs). With credentials it's broken.
    - 'null' origin is sent by sandboxed iframes, file:// pages, and
      certain redirects — an easy attacker path.

[PT] PROPÓSITO
    Sonda um URL com vários headers Origin e inspeciona os headers de
    resposta Access-Control-Allow-*. Envia pedidos GET e OPTIONS
    (preflight) com origens de atacante arbitrárias, a origem 'null'
    mágica, truques de prefixo/sufixo
    (`target.example.attacker.com`, `attacker.com.target.example`),
    e uma origem legítima. Assinala origens refletidas, ACAO wildcard com
    credentials, origem 'null' aceite, e bugs de prefix/suffix matching
    ingénuos.

[PT] ARRANQUE RÁPIDO
    $ tools/cors_check.py https://api.target.example/users/me

[PT] EXEMPLOS
    # Cenário 1 — testar se um endpoint de API reflete origens arbitrárias
    $ tools/cors_check.py https://api.target.example/users/me
    Output esperado (truncado):
        Sondagens:
          Origin: https://attacker.example
            GET     → ACAO: https://attacker.example  ACAC: true   [✗ REFLETIDO]
            OPTIONS → ACAO: https://attacker.example  ACAM: GET,POST,DELETE
          Origin: null
            GET     → ACAO: null   ACAC: true                      [✗ NULL OK]
          ...
        Problemas encontrados (2):
          ✗ Origin arbitrário refletido com credentials
            Risco:    qualquer site pode ler a resposta com os cookies do
                      utilizador. CSRF + roubo de dados.
            Correção: validar Origin contra uma allowlist no servidor.

    # Cenário 2 — JSON para um relatório de recon
    $ tools/cors_check.py https://api.example.com/me --json

    # Cenário 3 — PT-PT para um writeup
    $ tools/cors_check.py https://api.example.com/me --lang pt

[PT] FLAGS
    Para além das universais: apenas --timeout SEGUNDOS (por omissão 10).

[PT] CÓDIGOS DE SAÍDA
    0  Sem comportamento CORS arriscado
    1  Origens refletidas, wildcard+creds, null aceite, etc.
    2  URL inválido
    3  Erro de rede

[PT] DICAS
    - Corre contra endpoints autenticados (depois de login) — má
      configuração de CORS é mais danosa quando o endpoint pede credenciais.
    - ACAO wildcard ('*') sem credentials é muitas vezes intencional e
      inofensivo (APIs públicas). Com credentials está partido.
    - Origin 'null' é enviado por iframes em sandbox, páginas file:// e
      certos redirects — caminho fácil para um atacante.

1.4 http_methods.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Test which HTTP methods (GET, HEAD, OPTIONS, POST, PUT, PATCH, DELETE,
    TRACE, CONNECT) an endpoint accepts. Flags TRACE/CONNECT enabled,
    destructive methods (PUT/DELETE/PATCH) returning 2xx without auth,
    OPTIONS missing the Allow header, and TRACE that reflects request
    headers (XST — Cross-Site Tracing).

    WARNING: sends real PUT/DELETE/PATCH with empty bodies. Use only on
    targets you are authorised to test. Never against production endpoints
    that perform real writes.

[EN] QUICK START
    $ tools/http_methods.py https://example.com/api/users/me

[EN] EXAMPLES
    # Scenario 1 — REST endpoint with broken auth
    $ tools/http_methods.py https://target.example/api/users/42
    Expected output (truncated):
        URL: https://target.example/api/users/42
        Methods tested: 9
          GET      → 200  (✓ expected)
          HEAD     → 200
          OPTIONS  → 200  Allow: GET, POST, PUT, DELETE
          POST     → 405
          PUT      → 200  [✗ destructive method returned 2xx]
          DELETE   → 204  [✗ destructive method returned 2xx]
          TRACE    → 200  [✗ TRACE enabled (XST risk)]
          CONNECT  → 405
        Issues found (3): ...

    # Scenario 2 — quick TRACE check on a legacy app
    $ tools/http_methods.py http://legacy.example/ --json | jq '.tested.TRACE'

    # Scenario 3 — PT-PT
    $ tools/http_methods.py https://example.com/ --lang pt

[EN] FLAGS
    Beyond the universals: only --timeout SECONDS (default 10).

[EN] EXIT CODES
    0  Only safe methods accepted, no risky combinations
    1  Dangerous methods open (TRACE, destructive 2xx, etc.)
    2  Bad URL
    3  Network error

[EN] TIPS
    - Confirm authorisation in writing before running this against any
      real target. Empty PUT to a wrong endpoint can blank a record.
    - TRACE + missing X-XSS-Protection used to be the classic XST chain.
      Modern browsers blocked it, but TRACE enabled is still a smell.
    - Pair with cookie_check: if Authorization cookies are HttpOnly but
      PUT/DELETE are accessible, the impact is limited to the local user.

[PT] PROPÓSITO
    Testa quais os métodos HTTP (GET, HEAD, OPTIONS, POST, PUT, PATCH,
    DELETE, TRACE, CONNECT) que um endpoint aceita. Assinala TRACE/CONNECT
    ativos, métodos destrutivos (PUT/DELETE/PATCH) com resposta 2xx sem
    autenticação, OPTIONS sem header Allow, e TRACE que reflete os
    headers do pedido (XST — Cross-Site Tracing).

    AVISO: envia pedidos PUT/DELETE/PATCH reais com corpos vazios. Usa
    apenas em alvos autorizados. Nunca contra endpoints de produção que
    fazem escritas reais.

[PT] ARRANQUE RÁPIDO
    $ tools/http_methods.py https://example.com/api/users/me

[PT] EXEMPLOS
    # Cenário 1 — endpoint REST com autenticação partida
    $ tools/http_methods.py https://target.example/api/users/42
    Output esperado (truncado):
        URL: https://target.example/api/users/42
        Métodos testados: 9
          GET      → 200  (✓ esperado)
          PUT      → 200  [✗ método destrutivo devolveu 2xx]
          DELETE   → 204  [✗ método destrutivo devolveu 2xx]
          TRACE    → 200  [✗ TRACE ativado (risco XST)]
          ...
        Problemas encontrados (3): ...

    # Cenário 2 — verificação rápida de TRACE numa aplicação legacy
    $ tools/http_methods.py http://legacy.example/ --json | jq '.tested.TRACE'

    # Cenário 3 — PT-PT
    $ tools/http_methods.py https://example.com/ --lang pt

[PT] FLAGS
    Para além das universais: apenas --timeout SEGUNDOS (por omissão 10).

[PT] CÓDIGOS DE SAÍDA
    0  Só métodos seguros aceites, sem combinações arriscadas
    1  Métodos perigosos abertos (TRACE, destrutivos com 2xx, etc.)
    2  URL inválido
    3  Erro de rede

[PT] DICAS
    - Confirma autorização por escrito antes de correr contra um alvo
      real. Um PUT vazio para o endpoint errado pode esvaziar um registo.
    - TRACE + ausência de X-XSS-Protection era a cadeia XST clássica.
      Os browsers modernos bloquearam-no, mas TRACE ativo continua a ser
      um indicador.
    - Combina com cookie_check: se cookies de Authorization são HttpOnly
      mas PUT/DELETE são acessíveis, o impacto fica limitado ao utilizador.

1.5 robots_check.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Fetch /robots.txt and /sitemap.xml, list per-User-Agent rules, and
    highlight paths that match interesting patterns (admin, api, backup,
    debug, dev, staging, .git, .env, internal, private, secret, etc.).
    Prints the first 50 sitemap URLs as a recon starting point. Often the
    fastest way to find the hidden admin panel that the site owner forgot
    they listed.

[EN] QUICK START
    $ tools/robots_check.py https://example.com

[EN] EXAMPLES
    # Scenario 1 — recon on a CTF target
    $ tools/robots_check.py https://target.example
    Expected output (truncated):
        URL: https://target.example/robots.txt
        Status: 200
        Rules:
          User-Agent: *
            Disallow: /admin/         [! interesting]
            Disallow: /api/v1/internal [! interesting]
            Disallow: /backup.zip     [! interesting]
            Disallow: /
          User-Agent: Googlebot
            Allow: /
        Sitemap: https://target.example/sitemap.xml (124 URLs)
          1. https://target.example/
          2. https://target.example/about
          ...

    # Scenario 2 — JSON for piping
    $ tools/robots_check.py https://example.com --json | jq '.interesting'

    # Scenario 3 — PT-PT
    $ tools/robots_check.py https://example.com --lang pt

[EN] FLAGS
    Beyond the universals: only --timeout SECONDS (default 10).

[EN] EXIT CODES
    0  robots.txt fetched, no interesting patterns
    1  Interesting paths found
    2  Bad URL
    3  Network error

[EN] TIPS
    - "Interesting" doesn't mean exploitable — Disallow on /admin/ tells
      you it exists. Whether it's accessible is a separate test.
    - sitemap.xml frequently leaks staging URLs and back-end domains.
    - Check the User-Agent rules: some sites serve different content to

1.10 param_miner.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Discover hidden or undocumented query parameters by fuzzing a wordlist
    of 278 common parameter names against a target URL. Establishes a
    response baseline (HTTP status + body length), then probes each name
    with a canary value and flags any parameter that produces a measurable
    difference: status code change, body length delta >= threshold, or the
    parameter value reflected in the response body. Useful for surfacing
    cache-poisoning vectors, mass-assignment bugs, internal feature flags,
    and forgotten debug parameters.

[EN] QUICK START
    $ tools/param_miner.py https://api.example.com/users

[EN] EXAMPLES
    # Scenario 1 — basic parameter discovery on a REST endpoint
    $ tools/param_miner.py https://api.example.com/users
    Expected output (truncated):
        Target: https://api.example.com/users
        Parameters tested: 278
        Baseline status: 200  |  Baseline response length: 1432 bytes

        Hidden parameters found (2):

          ✗ Parameter 'debug' detected
             Status changed: 200 → 200
             Length delta: 1432 → 2817 bytes (+1385)
             Risk: Hidden parameters may indicate overlooked functionality ...
             Fix:  Document all parameters your API accepts ...

    # Scenario 2 — raise the threshold to cut noise on a chatty endpoint
    $ tools/param_miner.py https://api.example.com/users --threshold 50

    # Scenario 3 — JSON output for pipeline use
    $ tools/param_miner.py https://api.example.com/users --json | \
        jq '.findings[] | {param, status, length}'

[EN] FLAGS
    Beyond the universals (--lang, --json, --version, --user-agent):
    --threshold BYTES    Minimum body length delta to flag. Default: 5.
                         Raise to 20-50 on endpoints with dynamic content.
    --threads N          Concurrent probes. Default: 10.
    --timeout SECS       Per-request timeout. Default: 10.
    --method GET|POST    HTTP method. Default: GET.
    --content-type form|json  POST body encoding. Default: form.
                         (only relevant with --method POST)
    --wordlist FILE      Custom wordlist (one parameter name per line).
                         Replaces the built-in 278-name list.

[EN] EXIT CODES
    0  No parameters detected beyond baseline
    1  One or more hidden parameters found
    2  Bad URL format or usage error
    3  Network / DNS / TLS error

[EN] TIPS
    - A length delta alone is weak evidence — verify manually that the
      delta persists on a second request (it may be dynamic content).
    - Reflected parameters (value appears in the body) are stronger signal
      and often point to mass-assignment or template-injection.
    - Combine with cors_check: a parameter that changes the response may
      also change CORS behaviour.
    - Default threshold of 5 bytes is sensitive but noisy on pages with
      timestamps or anti-CSRF tokens in the body. Raise to 50+ for those.

[PT] PROPÓSITO
    Descobre parâmetros de query ocultos ou não documentados testando
    uma wordlist de 278 nomes comuns contra um URL alvo. Estabelece uma
    baseline de resposta (status HTTP + tamanho do body), depois sonda
    cada nome com um valor canary e sinaliza qualquer parâmetro que
    produza uma diferença medível: mudança de status code, delta de
    tamanho >= limiar, ou o valor do parâmetro refletido no body.
    Útil para surfacing de vetores de cache poisoning, bugs de
    mass-assignment, feature flags internas, e parâmetros de debug
    esquecidos.

[PT] ARRANQUE RÁPIDO
    $ tools/param_miner.py https://api.example.com/users

[EN] PURPOSE

2.5 subdomain_takeover.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Detect subdomains vulnerable to takeover — where a CNAME points to
    a third-party service that is no longer registered (e.g. a forgotten
    GitHub Pages site, Heroku app, S3 bucket). An attacker can register
    that orphaned service and serve content under the original domain.
    Workflow: enumerate subdomains (crt.sh, or supply your own list),
    resolve CNAMEs via raw UDP DNS, match against 16 service patterns,
    confirm by HTTP-probing for the service's "unclaimed" fingerprint.

[EN] QUICK START

            Service: Heroku
            Status: VULNERABLE — unclaimed service detected

    1  At least one vulnerable subdomain detected
    2  Bad domain or unreadable subdomains file
    3  Network error (DNS resolver unreachable, crt.sh down)

3.1 tls_inspect.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Connect to a host:port and report everything about the TLS
    certificate and connection: issuer, subject, validity window, SANs,
    signature algorithm, public key size, TLS version negotiated, cipher
    suite. Fetches certs even when they are broken (uses CERT_NONE) so
    you can inspect expired or self-signed certs. Additionally probes
    which TLS versions the server accepts via pinned handshakes
    (SSLv3, TLS 1.0, 1.1, 1.2, 1.3) and flags the deprecated ones.

[EN] QUICK START
    $ tools/tls_inspect.py example.com

[EN] EXAMPLES
    # Scenario 1 — full cert inspection
    $ tools/tls_inspect.py target.example
    Expected output (truncated):
        Host: target.example:443
        Subject:    CN=target.example
        Issuer:     CN=R3, O=Let's Encrypt
        Valid from: 2026-01-15  to  2026-04-15  (✗ expires in 5 days)
        SANs:       target.example, www.target.example
        Sig algo:   sha256WithRSAEncryption
        Public key: RSA 2048 bits
        Negotiated: TLSv1.3  TLS_AES_256_GCM_SHA384
        Accepted:   TLS 1.2, TLS 1.3
        Refused:    SSLv3, TLS 1.0, TLS 1.1     ✓

    # Scenario 2 — non-default port
    $ tools/tls_inspect.py mail.example.com:993

    # Scenario 3 — legacy server (e.g. an internal box stuck on TLS 1.0)
    $ tools/tls_inspect.py legacy.example.local --lang pt

[EN] FLAGS
    Beyond the universals: only --timeout SECONDS (default 10).
    Target accepts plain `host`, `host:port`, or a full URL.

[EN] EXIT CODES
    0  Cert valid, no weak/deprecated TLS versions
    1  Expired, expiring soon, weak sig, hostname mismatch, broad
       wildcard, or deprecated TLS version accepted
    2  Bad target format
    3  Connection or TLS error

[EN] TIPS
    - 5-day expiry warning is intentional — it's enough lead time to
      renew before users notice.
    - "Accepted: TLS 1.0" is a real finding even if the cert is fine —
      old protocols expose old ciphers.
    - Internal IPs and self-signed certs work — the tool doesn't require
      a valid chain to inspect. Useful for AD lab work.

[PT] PROPÓSITO
    Liga-se a um host:port e reporta tudo sobre o certificado TLS e a
    ligação: issuer, subject, janela de validade, SANs, algoritmo de
    assinatura, tamanho da chave pública, versão de TLS negociada, suite
    de cifras. Vai buscar certificados mesmo quando estão partidos (usa
    CERT_NONE) para se poder inspecionar certs expirados ou

4.1 jwt_inspect.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Decode a JSON Web Token into its three parts (header, payload,
    signature), pretty-print the claims with human-readable timestamps
    and validity windows, and flag common security issues: `alg: none`,
    unknown/weak algorithms, missing or expired `exp`, future `nbf`,
    validity windows >24h, missing `iss`/`aud`, and path-traversal
    patterns in the `kid` header.

[EN] QUICK START
    $ tools/jwt_inspect.py "eyJhbGciOi..."

[EN] EXAMPLES
    # Scenario 1 — quick inspection of a captured token
    $ tools/jwt_inspect.py "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    Expected output (truncated):
        Header:
          alg: HS256
          typ: JWT
        Payload:
          sub: 42
          name: Alice
          iat: 1714572345 (2024-05-01 14:25:45 UTC)
          exp: 1715177145 (2024-05-08 14:25:45 UTC) [✗ expired]
        Signature: 32-byte HMAC-SHA256
        Issues:
          ✗ Token expired since 2024-05-08

    # Scenario 2 — read JWT from stdin
    $ echo "$JWT" | tools/jwt_inspect.py -

    # Scenario 3 — PT-PT for a writeup
    $ tools/jwt_inspect.py "$JWT" --lang pt --json

[EN] FLAGS
    Beyond the universals: none.

[EN] EXIT CODES
    0  No issues
    1  Issues found (expired, alg:none, etc.)
    2  Not a valid JWT format (not 3 base64url segments)

[EN] TIPS
    - `alg: none` tokens accept any signature (including empty). Many
      libraries used to allow this by default — still occasionally seen.
    - `kid` (key ID) header is a free-text field used by the server to
      pick the verification key. If the server uses `kid` to read a file
      path, traversal like `../../../../../dev/null` can force a
      predictable empty key.
    - Don't paste real production tokens into public terminals — they
      grant whatever access the `sub` claim implies until `exp`.

[PT] PROPÓSITO
    Descodifica um JWT nas três partes (header, payload, signature),
    imprime as claims com timestamps legíveis e janelas de validade, e
    assinala problemas comuns de segurança: `alg: none`, algoritmos
    desconhecidos/fracos, `exp` em falta ou expirado, `nbf` no futuro,
    janelas de validade > 24h, `iss`/`aud` em falta, e padrões de path
    traversal no header `kid`.

[PT] ARRANQUE RÁPIDO
    $ tools/jwt_inspect.py "eyJhbGciOi..."

[PT] EXEMPLOS

          ✗ Token expirado desde 2024-05-08

    # Cenário 2 — ler JWT de stdin
    $ echo "$JWT" | tools/jwt_inspect.py -

    # Cenário 3 — PT-PT para um writeup
    $ tools/jwt_inspect.py "$JWT" --lang pt --json

[PT] FLAGS
    Para além das universais: nenhuma.

[PT] CÓDIGOS DE SAÍDA
    0  Sem problemas
    1  Problemas encontrados (expirado, alg:none, etc.)
    2  Não é um JWT válido (não tem 3 segmentos base64url)

[PT] DICAS

    - O header `kid` (key ID) é um campo de texto livre usado pelo

      acesso que a claim `sub` implica até ao `exp`.

4.2 hashid.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Identify the likely format(s) of a hash string. Knows ~25 formats:
    MD5, SHA-1/256/384/512, bcrypt ($2a/$2b/$2y$...), scrypt, Argon2
    ($argon2id$...), phpass (WordPress), Apache APR1 ($apr1$...), Unix
    crypt variants ($1$/$5$/$6$/$y$), MySQL old (16 hex) and new (* +
    40 hex), NTLM (32 hex, no salt), LM, LDAP (SHA/SSHA base64), JWT
    (commonly mistaken for a hash). Ranks candidates by confidence and
    includes hashcat mode hints.

[EN] QUICK START
    $ tools/hashid.py 5f4dcc3b5aa765d61d8327deb882cf99

[EN] EXAMPLES
    # Scenario 1 — anonymous hash from a CTF
    $ tools/hashid.py 5f4dcc3b5aa765d61d8327deb882cf99
    Expected output:
        Input: 5f4dcc3b5aa765d61d8327deb882cf99 (32 hex chars)
        Candidates:
          1. MD5         confidence: high    hashcat -m 0
          2. NTLM        confidence: high    hashcat -m 1000
          3. MD4         confidence: medium  hashcat -m 900
          4. LM          confidence: low     hashcat -m 3000

    # Scenario 2 — bcrypt hash
    $ tools/hashid.py '$2b$12$LQv3c1yqBWVHxkd0LHAkCOYz...'
    Expected output:
        Candidates: bcrypt (high, hashcat -m 3200)

    # Scenario 3 — from stdin (useful with `head /etc/shadow`)
    $ echo '$6$saltsaltsalt$...' | tools/hashid.py -

[EN] FLAGS
    Beyond the universals: none.

[EN] EXIT CODES
    0  At least one candidate identified
    2  Empty input or invalid format

[EN] TIPS
    - 32 hex chars is ambiguous: MD5 vs NTLM is the most common
      confusion. Context decides: shadow file → MD5-crypt; SAM dump →

      crack with hashcat -m 3200.

      `hashcat -m 1000 hash.txt rockyou.txt`.

[PT] PROPÓSITO

[PT] FLAGS
    Para além das universais: nenhuma.

[PT] CÓDIGOS DE SAÍDA

[PT] DICAS

5.1 password_strength.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Analyze password strength locally (length, character classes,
    entropy bits, 0-10 score) and check whether the password appears in
    the Have I Been Pwned breach corpus. The HIBP check uses the
    k-anonymity API: only the first 5 chars of the SHA-1 hash are sent
    over the wire; the full password never leaves the machine.

[EN] QUICK START
    $ tools/password_strength.py            # interactive prompt, no echo

[EN] EXAMPLES
    # Scenario 1 — interactive (default)
    $ tools/password_strength.py
    Password: (typed silently)
    Expected output:
        Length:    12
        Charsets:  lower upper digit symbol  (4 classes)
        Entropy:   78.4 bits
        Score:     8/10 (strong)
        HIBP:      ✓ not in breach corpus

    # Scenario 2 — from a script (no terminal)
    $ echo -n "hunter2" | tools/password_strength.py --stdin
    Expected output:
        Length:    7
        Charsets:  lower digit  (2 classes)
        Entropy:   33.7 bits
        Score:     2/10 (very weak)
        HIBP:      ✗ found in 19383 breaches

    # Scenario 3 — score only, skip the network call
    $ echo -n "hunter2" | tools/password_strength.py --stdin --no-breach \
        --lang pt --json

[EN] FLAGS
    Beyond the universals (--lang, --json, --version, --user-agent):
    --stdin             Read from stdin (otherwise prompts interactively).
    --no-breach         Skip the HIBP check (no network).
    --timeout SECONDS   HIBP request timeout. Default 10.

[EN] EXIT CODES
    0  Not in breach corpus (or --no-breach used)
    1  Found in HIBP corpus
    2  Empty password

[EN] TIPS
    - k-anonymity means HIBP only sees the first 5 SHA-1 hex chars (one
      bucket of ~500 hashes). It returns the full bucket and the tool
      filters locally for the match.
    - Score caps at 10 — anything past ~80 bits of entropy already
      maxes out. Use as a sanity check, not an oracle.
    - For password generation, use a passphrase manager. This tool is
      for checking, not generating.

[PT] PROPÓSITO

    caracteres, bits de entropia, pontuação 0-10) e verifica se a
    password aparece no corpus de breaches Have I Been Pwned. A
    verificação HIBP usa a API de k-anonymity: só os primeiros 5
    caracteres do SHA-1 são enviados pela rede; a password completa
    nunca sai da máquina.

5.2 secrets_scan.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Scan a filesystem tree or a git history for committed credentials.
    Filesystem mode walks the path (skipping binaries and common noise:
    node_modules, .venv, __pycache__, etc.). Git mode runs `git log -p`
    and inspects every diff hunk across history. Built-in patterns cover
    AWS access keys, Stripe / Slack / GitHub / Google / Twilio /
    SendGrid tokens, private keys (RSA/EC/PEM), JWT strings, .env-style
    assignments, and database URLs. Generic `KEY = "value"` assignments
    are gated by a Shannon-entropy threshold (≥ 3.5 bits/char) to cut
    false positives.

[EN] QUICK START
    $ tools/secrets_scan.py                  # current directory
    $ tools/secrets_scan.py --path ~/Projects/some-repo
    $ tools/secrets_scan.py --git-history    # all commits

[EN] EXAMPLES
    # Scenario 1 — quick audit before publishing a repo
    $ cd ~/Projects/my-project
    $ tools/secrets_scan.py
    Expected output (truncated):
        Scanning: .
        Findings (3):
          src/config.py:18
            pattern: aws_access_key_id
            match:   AKIAIOSFODNN7EXAMPLE
          tests/fixtures.json:42
            pattern: stripe_secret_key
            match:   sk_live_4eC39HqLyjWDarjtT...
          .env.local:5
            pattern: generic_secret (entropy 4.8 bits/char)
            match:   API_TOKEN="aB3xJ9..."

    # Scenario 2 — scan a different repo
    $ tools/secrets_scan.py --path /tmp/cloned-target

    # Scenario 3 — historical scan (commits + diffs)
    $ tools/secrets_scan.py --git-history --lang pt --json

[EN] FLAGS
    Beyond the universals (--lang, --json, --version):
    --path DIR          Directory to scan. Default: current dir.
    --git-history       Scan `git log -p` (all commits) instead of the
                        working tree.

[EN] EXIT CODES
    0  No secrets found
    1  Findings present
    2  Bad path, or --git-history used outside a git repo

[EN] TIPS
    - --git-history scans everything, including files now deleted —
      that's the whole point. A leaked secret committed once is leaked
      forever until you rotate it.
    - Rotate the secret first, redact later. `git filter-repo` or
      `git filter-branch` can rewrite history, but you also have to
      force-push and coordinate with collaborators.

      for high-entropy non-secret strings (UUIDs, large hashes). Review
      manually before action.

    $ tools/secrets_scan.py --git-history     # todos os commits

6.1 multidecode.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Auto-detect and decode common encodings: Base64, Base32, Base85,
    hex, URL percent-encoding, HTML entities, Unicode escapes (\uXXXX,
    \u{XXXX}, \xXX), binary (0/1 strings), and ROT13. Useful when a
    CTF drops a suspicious string in your lap or you see a payload in
    HTTP traffic. `--cascade` recursively decodes until the result
    stops looking encoded (e.g. Base64 of Base64 of hex → plaintext).

[EN] QUICK START
    $ tools/multidecode.py "SGVsbG8gd29ybGQh"

[EN] EXAMPLES

    $ tools/multidecode.py "SGVsbG8gd29ybGQh"
    Expected output:
        Input: 'SGVsbG8gd29ybGQh' (length: 16)

        Decodings that produced readable output:

          [Base64]
            'Hello world!'

    # Scenario 2 — Unicode escape payload from a JS source
    $ tools/multidecode.py '\x48\x65\x6c\x6c\x6f'
    Expected output:
        [Unicode Escapes]
          'Hello'

    # Scenario 3 — Base85 (common in PDF / PostScript)
    $ python3 -c "import base64; print(base64.b85encode(b'Hello World').decode())"
    NM&qnZy;B1a%^M
    $ tools/multidecode.py "NM&qnZy;B1a%^M"
    Expected output:
        [Base85]
          'Hello World'

    # Scenario 4 — nested encoding (Base64 of Base64)
    $ tools/multidecode.py "VkdobElHSnliM2R1SUdadmVB" --cascade
    Expected output: [depth 1] Base64 → [depth 2] Base64 → "The brown fox"

[EN] FLAGS
    Beyond the universals (--lang, --json, --version):

                        ROT13 is excluded from cascade (it is its own
                        inverse and would loop).

[EN] EXIT CODES
    0  At least one decoding succeeded
    2  Empty input
    3  Stdin error

    - For ambiguous strings, look at the output of all candidates, not

6.2 xor_crack.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    Recover plaintext from XOR-encrypted ciphertext using English letter-
    frequency analysis. Single-byte mode brute-forces all 256 keys and
    ranks candidates by English score (Cryptopals challenge #3 in one
    command). Multi-byte mode uses the Index of Coincidence to guess
    the key length, then solves each key byte independently
    (Cryptopals challenge #6). Input is hex by default; `--raw` reads
    raw binary from stdin.

[EN] QUICK START
    $ tools/xor_crack.py 1b37373331363f78151b7f2b783431333d78397828372d363c78373e783a393b3736

[EN] EXAMPLES
    # Scenario 1 — single-byte XOR (Cryptopals #3)
    $ tools/xor_crack.py 1b37373331363f78151b7f2b78343133...
    Expected output:
        Single-byte XOR — top candidates (34 bytes):
          0x58 ('X')  score +6.35  :  Cooking MC's like a pound of bacon
          0x78 ('x')  score +6.33  :  cOOKING MC'S LIKE A POUND OF BACON
          0x52 ('R')  score +4.46  :  Ieeacdm*GI ...
          ...

    # Scenario 2 — multi-byte with auto-detection (Cryptopals #6 excerpt)
    $ tools/xor_crack.py <hex-of-ciphertext>
    Expected output:
        Index of Coincidence by key length:
          klen 3: IC = 0.0421
          klen 6: IC = 0.0658  ← best
          klen 9: IC = 0.0418
          ...
        Best key length: 6
        Recovered key: 49434549415f4d (hex) / "ICEMAN_M" (ascii?)
        Plaintext: I'm back and I'm ringin' the bell ...

    # Scenario 3 — known key length (force it)
    $ tools/xor_crack.py <hex> --key-len 3

    # Scenario 4 — decrypt with a known key
    $ tools/xor_crack.py <hex> --key 6b6579

    # Scenario 5 — raw binary input from stdin
    $ cat cipher.bin | tools/xor_crack.py - --raw

[EN] FLAGS
    Beyond the universals (--lang, --json, --version):
    --raw           Read raw bytes from stdin (use with '-').
    --key-len N     Force key length N (skip auto-detection).
    --max-len N     Max key length to consider. Default 16.
    --top TOP       Number of top candidates for single-byte. Default 5.
    --key HEX       Decrypt with a known key (hex) — skip cracking.

[EN] EXIT CODES
    0  Cracking attempted (regardless of how "real" the plaintext looks)
    2  Invalid hex or empty input

[EN] TIPS
    - English frequency analysis assumes English plaintext. For other
      languages, the scoring will mislead — be ready to spot it.
    - IC is a heuristic. The best klen is usually right but not always.
      Try the top 3 candidates if the plaintext looks wrong.
    - For one-time-pad (key length == plaintext length), XOR is
      uncrackable. The IC table will be flat at 0.038 (random).

[PT] PROPÓSITO

    frequência de letras em inglês. O modo de byte único faz
    brute-force a todas as 256 chaves e ordena candidatos por
    pontuação inglesa (Cryptopals challenge #3 num comando). O modo
    multi-byte usa o Índice de Coincidência para adivinhar o
    comprimento da chave, depois resolve cada byte da chave
    independentemente (Cryptopals challenge #6). Input é hex por
    omissão; `--raw` lê bytes binários de stdin.

[PT] ARRANQUE RÁPIDO
    $ tools/xor_crack.py 1b37373331363f78151b7f2b78343133...

[PT] EXEMPLOS
    # Cenário 1 — XOR de byte único (Cryptopals #3)
    $ tools/xor_crack.py 1b37373331363f78151b7f2b78343133...
    Output esperado:
        XOR de byte único — melhores candidatos (34 bytes):
          0x58 ('X')  pontuação +6.35  :  Cooking MC's like a pound of bacon
          0x78 ('x')  pontuação +6.33  :  cOOKING MC'S LIKE A POUND OF BACON

    # Cenário 2 — multi-byte com auto-deteção (excerto Cryptopals #6)

    Output esperado:
        Índice de Coincidência por tamanho de chave:

    $ tools/file_hash.py installer.iso \
        --verify 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73...

[EN] FLAGS
    Beyond the universals (--lang, --json, --version):
    --algos LIST   Comma-separated algorithms. Default
                   md5,sha1,sha256,sha512. Any name accepted by
                   hashlib.new() works (sha3_256, blake2b, etc.).
    --verify HASH  Compare with this hash. Uses the first --algos.
                   Mutually exclusive with --save and --check.
    --save FILE    Write a GNU coreutils-style manifest (<hash>  <name>).
                   Mutually exclusive with --verify and --check.
    --check FILE   Verify files listed in a manifest. Mutually exclusive
                   with --verify and --save.

[EN] EXIT CODES
    0  All hashes computed (default) / match (verify) / all OK (check)
    1  Mismatch (verify or check) or any check entry failed
    2  Usage error (file not found, invalid algorithm, IO error)

[EN] TIPS
    - Always hash evidence BEFORE working on it. The hash establishes
      that downstream analysis touched the same bytes.
    - Manifest format is interoperable with GNU coreutils — you can
      verify with `sha256sum -c manifest.sha256` on Linux too.
    - For very large files (multi-GB), the streaming I/O is what makes
      this safe to run on a workstation with limited RAM.

[PT] PROPÓSITO
    Hashing forense de ficheiros para cadeia de custódia (chain-of-
    custody). Calcula MD5, SHA-1, SHA-256, SHA-512 (mais SHA-3 e
    variantes BLAKE2) numa só passagem com chunks de 64 KB em streaming
    — aguenta ficheiros de qualquer tamanho sem pressão de memória.
    Quatro modos:

      - default        Imprime hashes de um ou mais ficheiros
      - --verify HASH  Compara contra um hash esperado (exit 0/1)
      - --save FILE    Escreve um manifesto estilo GNU
      - --check FILE   Verifica os ficheiros listados num manifesto,
                       reporta OK / NÃO CORRESPONDE / EM FALTA por entrada

    Inspirado pela unidade UC01489 (recolha e análise forense digital
    — integridade da prova) do currículo CET português de Técnico
    Especialista em Cibersegurança.

[PT] ARRANQUE RÁPIDO
    $ tools/file_hash.py evidence.bin

[PT] EXEMPLOS
    # Cenário 1 — multi-hash a um ficheiro de prova
    $ tools/file_hash.py evidence.bin --lang pt
    Output esperado (truncado):
        Ficheiro: evidence.bin
        Tamanho:  1.234.567 bytes (1.18 MiB)
          MD5:     5d41402abc4b2a76b9719d911017c592
          SHA-1:   aaf4c61ddcc5e8a2dabede0f3b482cd9aea9434d
          SHA-256: 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73...

    # Cenário 2 — guardar e mais tarde verificar um manifesto
    $ tools/file_hash.py *.bin --save manifest.sha256 --lang pt
    Manifesto guardado: manifest.sha256 (4 ficheiros, sha256)
    $ tools/file_hash.py --check manifest.sha256 --lang pt
        ✓ evidence.bin
        ✗ tampered.bin   [NÃO CORRESPONDE — esperado ..., obtido ...]
        ! missing.bin    [FICHEIRO NÃO ENCONTRADO]

    # Cenário 3 — verificar um download contra um hash publicado

[PT] FLAGS
    Para além das universais (--lang, --json, --version):
    --algos LISTA   Algoritmos separados por vírgula. Por omissão
                    md5,sha1,sha256,sha512. Qualquer nome aceite por
                    hashlib.new() funciona (sha3_256, blake2b, etc.).
    --verify HASH   Compara com este hash. Usa o primeiro de --algos.
                    Mutuamente exclusivo com --save e --check.
    --save FICHEIRO Escreve um manifesto estilo GNU coreutils
                    (<hash>  <nome>). Mutuamente exclusivo com
                    --verify e --check.
    --check FICHEIRO Verifica os ficheiros listados num manifesto.
                    Mutuamente exclusivo com --verify e --save.

[PT] CÓDIGOS DE SAÍDA
    0  Todos os hashes calculados (default) / match (verify) / tudo OK (check)
    1  Mismatch (verify ou check) ou alguma entrada do check falhou
    2  Erro de uso (ficheiro inexistente, algoritmo inválido, erro de IO)

[PT] DICAS
    - Faz sempre hash da prova ANTES de trabalhar nela. O hash
      estabelece que a análise downstream tocou nos mesmos bytes.
    - O formato do manifesto é interoperável com o GNU coreutils —
      podes também verificar com `sha256sum -c manifest.sha256` em Linux.
    - Para ficheiros muito grandes (vários GB), o I/O em streaming é
      o que torna seguro correr numa workstation com pouca RAM.

8.1 recon.py
--------------------------------------------------------------------------------

[EN] PURPOSE
    One-command reconnaissance orchestrator that composes the other
    tools. Workflow: (1) run subfinder to enumerate subdomains;
    (2) for each resolved subdomain (top N) in parallel, run
    check_headers + tls_inspect + cookie_check; (3) run dns_records
    against the base domain; (4) aggregate everything into a single
    Markdown report. Use it as the starting point of any deep-dive on
    a target, then drill in with individual tools where needed.

[EN] QUICK START
    $ tools/recon.py example.com

[EN] EXAMPLES
    # Scenario 1 — quick recon on a bug bounty target
    $ tools/recon.py target.example --top 10
    Expected output: Markdown report on stdout with sections per
    subdomain, summarising header status, TLS findings, and cookie
    issues. ~200-500 lines depending on subdomain count.

    # Scenario 2 — write report to a file
    $ tools/recon.py target.example --top 20 \
        --output target-recon-$(date +%F).md

    # Scenario 3 — PT-PT report for a writeup
    $ tools/recon.py target.example --lang pt --output recon.md

    # Scenario 4 — raw JSON output for programmatic processing
    $ tools/recon.py target.example --json > recon.json

[EN] FLAGS
    Beyond the universals (--lang, --json, --version):
    --top N             Max subdomains to scan per host. Default 10.
    --threads N         Parallel per-host scans. Default 6.
    --output FILE, -o   Write Markdown to a file instead of stdout.

[EN] EXIT CODES
    0  Recon completed
    2  Bad domain
    3  Network error or sibling tool failure

[EN] TIPS
    - Default --top 10 is intentionally low. For real recon, bump to
      --top 50 and be patient (a few minutes).
    - Each per-host scan is independent and can fail without aborting
      the whole report — failures are noted inline.
    - The Markdown report is a good starting draft for a writeup: open
      it, replace recon.py's prose with your own commentary, keep the
      tables.

[PT] PROPÓSITO
    Orquestrador de reconhecimento que compõe os outros tools num
    comando. Fluxo: (1) corre subfinder para enumerar subdomínios;
    (2) para cada subdomínio resolvido (top N) em paralelo, corre
    check_headers + tls_inspect + cookie_check; (3) corre dns_records
    contra o domínio base; (4) agrega tudo num relatório Markdown
    único. Usa-o como ponto de partida de qualquer deep-dive a um
    alvo, depois aprofunda com tools individuais onde necessário.

[PT] ARRANQUE RÁPIDO
    $ tools/recon.py example.com

[PT] EXEMPLOS
    # Cenário 1 — recon rápido a um alvo de bug bounty
    $ tools/recon.py target.example --top 10
    Output esperado: relatório Markdown em stdout com secções por
    subdomínio, a sumariar estado dos headers, findings TLS, e
    problemas de cookies. ~200-500 linhas dependendo do número de
    subdomínios.

    # Cenário 2 — escrever relatório para ficheiro
    $ tools/recon.py target.example --top 20 \
        --output target-recon-$(date +%F).md

    # Cenário 3 — relatório PT-PT para um writeup
    $ tools/recon.py target.example --lang pt --output recon.md

    # Cenário 4 — output JSON cru para processamento programático
    $ tools/recon.py target.example --json > recon.json

[PT] FLAGS
    Para além das universais (--lang, --json, --version):
    --top N             Subdomínios máximos a analisar por host. Por omissão 10.
    --threads N         Scans paralelos por host. Por omissão 6.
    --output FICHEIRO, -o   Escreve Markdown para ficheiro em vez de stdout.

[PT] CÓDIGOS DE SAÍDA
    0  Recon completo
    2  Domínio inválido
    3  Erro de rede ou falha de um tool irmão

[PT] DICAS
    - O --top 10 por omissão é intencionalmente baixo. Para recon a
      sério, sobe para --top 50 e tem paciência (alguns minutos).
    - Cada scan por host é independente e pode falhar sem abortar o
      relatório todo — as falhas são anotadas inline.
    - O relatório Markdown é um bom rascunho inicial para um writeup:
      abre-o, substitui a prosa do recon.py por comentário teu, mantém
      as tabelas.

9.1 htb_stats.py

[EN] PURPOSE
    Generate the Markdown for a HackTheBox profile badge, ready to paste
    into a README or About page (no API token required for the badge
    itself). Optionally fetch live profile stats — rank, points, machine
    owns, respects — via the HTB v4 API when an API token is set via
    `--token` or the `HTB_TOKEN` environment variable.

[EN] QUICK START
    $ tools/htb_stats.py 12345         # your HTB user ID (numeric)

[EN] EXAMPLES
    # Scenario 1 — badge only (no token needed)
    $ tools/htb_stats.py 12345 --badge-only
    Expected output:
        [![HTB Profile](https://www.hackthebox.com/badge/image/12345)
        ](https://app.hackthebox.com/users/12345)

    # Scenario 2 — full stats with API token
    $ HTB_TOKEN=eyJ0eXAiOi... tools/htb_stats.py 12345
    Expected output (truncated):
        Username:  Luis
        Rank:      Hacker
        Points:    142
        Owns:      User 18 / System 12
        Respects:  4
        Markdown:  [![HTB Profile]...

    # Scenario 3 — PT-PT, JSON
    $ tools/htb_stats.py 12345 --badge-only --lang pt --json

[EN] FLAGS
    Beyond the universals (--lang, --json, --version, --user-agent):
    --token TOKEN       HTB API bearer token (or set HTB_TOKEN env).
    --badge-only        Skip the API call, print badge markdown only.
    --timeout SECONDS   Default 10.

[EN] EXIT CODES
    0  Badge generated (and stats fetched if token provided)
    1  Stats fetch failed (network or auth issue)
    2  Non-numeric user ID
    3  API error

[EN] TIPS
    - The badge URL works without auth — that's what makes the
      --badge-only path useful even when you don't yet have an account
      with profile data.
    - HTB tokens are tied to your account. Don't commit them. Use the
      env var or a local .env, not a CLI flag in shell history.
    - HTB API changes break the stats fetch sometimes. The badge URL is
      stable.

[PT] PROPÓSITO
    Gera o Markdown para um badge de perfil HackTheBox, pronto para
    colar num README ou página About (sem necessidade de API token para
    o badge em si). Opcionalmente vai buscar estatísticas em direto do
    perfil — rank, pontos, máquinas conquistadas, respects — via a API
    HTB v4 quando um API token é definido com `--token` ou a env var
    `HTB_TOKEN`.

[PT] ARRANQUE RÁPIDO
    $ tools/htb_stats.py 12345         # o teu HTB user ID (numérico)

[PT] EXEMPLOS
    # Cenário 1 — apenas badge (sem token)

        Owns:       User 18 / System 12
        Respects:   4

[PT] FLAGS
    Para além das universais (--lang, --json, --version, --user-agent):
    --token TOKEN       API bearer token HTB (ou env HTB_TOKEN).
    --badge-only        Salta a chamada API, imprime só o badge.
    --timeout SEGUNDOS  Por omissão 10.

[PT] CÓDIGOS DE SAÍDA
    0  Badge gerado (e estatísticas se token fornecido)
    1  Falha ao obter estatísticas (rede ou auth)

        --type bugbounty --platform other --tags xss,web
